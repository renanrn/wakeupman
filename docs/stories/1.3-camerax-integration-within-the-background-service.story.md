# Story 1.3: CameraX integration within the background service

## Status
Approved

## Story
**As a** System,
**I want** to access camera frames without a UI preview,
**so that** ML Kit can process them efficiently in the background.

## Acceptance Criteria
1. `CameraX` library dependencies added to the project.
2. Custom `LifecycleOwner` implemented for `VigilanceService` (since standard Services lack a native lifecycle for CameraX binding).
3. `ProcessCameraProvider` initialized and bound to the service lifecycle.
4. `ImageAnalysis` use case configured to capture frames in `YUV_420_888` format.
5. Headless camera operation verified (no `Preview` use case required).
6. Basic `Analyzer` callback structure created to receive frames and close them (avoiding memory leaks).

## 🤖 CodeRabbit Integration
> **CodeRabbit Integration**: Disabled
>
> CodeRabbit CLI is not enabled in `core-config.yaml`.
> Quality validation will use manual review process only.
> To enable, set `coderabbit_integration.enabled: true` in core-config.yaml

## Tasks / Subtasks
- [ ] Task 1: Add CameraX dependencies (`camera-core`, `camera-camera2`, `camera-lifecycle`) to `app/build.gradle.kts`.
- [ ] Task 2: Create a custom `LifecycleOwner` implementation inside `VigilanceService` (using `LifecycleRegistry`).
- [ ] Task 3: Initialize `ProcessCameraProvider` via `ProcessCameraProvider.getInstance(context)`.
- [ ] Task 4: Configure `ImageAnalysis` use case (set backpressure strategy to `STRATEGY_KEEP_ONLY_LATEST`, format to `YUV_420_888`).
- [ ] Task 5: Bind the `ImageAnalysis` use case to the custom lifecycle using the front-facing camera.
- [ ] Task 6: Implement a placeholder `ImageAnalysis.Analyzer` that safely closes the `ImageProxy` after receiving it.

## Dev Notes
- Refer to `docs/architecture.md` Section 4.2 for Image Processing Pipeline details.
- Since this runs in a Foreground Service without an Activity, a `LifecycleService` or custom `LifecycleRegistry` is mandatory to bind CameraX.
- Ensure the `ImageProxy` is always closed in the analyzer, otherwise CameraX will block and stop emitting frames (memory leak check).
- Target framerate management will be fully handled in Epic 2, but keep performance in mind.

### Testing
- [ ] Verify that the service can start the camera without crashing.
- [ ] Log output inside the Analyzer callback to prove frames are being delivered continuously while the service runs.
- [ ] Verify that no memory leaks occur from unclosed `ImageProxy` instances over a 5-minute run.
